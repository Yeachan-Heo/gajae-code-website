Hermes / OpenClaw MCP Bridge

How a Hermes- or OpenClaw-style coordinator drives Gajae Code. The recommended starting point is the gjc tmux skill-based routed session; the GJC-native coordinator MCP bridge and the host-facing gjc-delegation skill share one core contract for pure machine control, and an RPC host-tool boundary covers the reverse direction.

Pick an integration path

  • Visible routed-session (gjc tmux, skill-based) — Recommended for starters drive a human-visible tmux pane with gjc-session helper scripts and inject /skill: prompts. The simplest, most observable way to start. Jump to routed sessions →
  • Coordinator MCP bridge — run gjc mcp-serve coordinator and let an external coordinator register sessions, send bounded turns, answer questions, and write reports. Jump to MCP bridge →
  • gjc-delegation skill (plugin) — install the generated gajae-code plugin so a host agent can delegate whole workflows (plan/execute/team) with one tool call. Jump to skill →
  • RPC host tools — when Hermes/OpenClaw needs GJC to call its tools, MCP servers, and skills, expose them as RPC customTools instead of importing GJC's MCP internals. Jump to RPC boundary →

Core contract and adapters

The coordinator bridge is intentionally a core contract with multiple adapters, not an MCP-only or Hermes-only product direction. Hermes and OpenClaw are compatibility presets, not privileged integration modes.

  • packages/coding-agent/src/coordinator/contract.ts owns transport-neutral server metadata and tool names.
  • gjc mcp-serve coordinator is the outward MCP adapter for external agents.
  • gjc coordinator is the read-only CLI/debug adapter for humans and scripts that need to inspect the same contract without starting MCP transport.
  • gjc setup hermes is the compatibility setup adapter that renders coordinator config and operator guidance.

This bridge is separate from GJC's client-side MCP runtime. gjc mcp-serve hermes is accepted as a compatibility alias for the same coordinator bridge.

Path 1 · Visible routed-session (gjc tmux, skill-based) Recommended for starters

This is the recommended way to get going. When a human or chatops router (Clawhip, Hermes, OpenClaw) must watch the work, receive stale-session alerts, and inject follow-up prompts into the same visible pane, drive an interactive gjc session in a routed tmux pane and inject /skill: prompts — rather than inventing a private terminal protocol. It is the simplest, most observable lane. Move to the Coordinator MCP bridge (Path 2/3) once you need pure machine control and durable turn state.

The public gjc-session plugin skill publishes this helper flow for operator repos. It is not a fifth bundled workflow skill; Gajae Code still has four core workflow skills. The published helpers are:

# create — register a routed tmux session and start interactive gjc in the worktree scripts/gjc-session/create.sh <session-name> <worktree-path> # prompt — inject the real task after the TUI is ready scripts/gjc-session/prompt.sh <session-name> @task.md # tail — inspect bounded pane output before/after prompt delivery scripts/gjc-session/tail.sh <session-name> 80 # owner starter used by create; do not wrap the owner in a short timeout scripts/gjc-session/harness-tmux-owner-start.sh

Contract: (1) create or verify a dedicated worktree, (2) register a stable named tmux session with the router before launching GJC, (3) start interactive gjc in the worktree, (4) wait for TUI readiness, (5) inject the task prompt separately, (6) verify acceptance from real work evidence — not from a visible pasted prompt. Use stable public-safe names that include project and artifact id (e.g. gajae-code-issue-905-ctrl-shift-enter-newline).

The injected prompt is skill-based, e.g.:

/skill:ralplan gjc ultragoal fix issue #905 missed Ctrl+Shift+Enter newline case. Repo: Yeachan-Heo/gajae-code Worktree: ../worktrees/gajae-code-issue-905-ctrl-shift-enter-newline Branch: issue-905-ctrl-shift-enter-newline Base: dev Scope: - inspect parser/key matching and the editor component - add explicit ctrl+shift+enter newline handling - add focused tests, run targeted verification - commit, push, and open a PR to dev Non-goals: - no unrelated tmux/session/process changes
Acceptance is evidence, not scrollback. Before reporting the session is working, require a real signal: a tool call/file read in the pane, an explicit plan/todo update, a diff or test command, or a GitHub comment/review/PR URL. A prompt visible in tmux scrollback is not acceptance. Do not launch visible routed work in the canonical checkout, and do not run a long session under a short shell timeout that can SIGKILL the owner.

Path 2 · Coordinator MCP bridge

For pure machine control with durable turn state, the bridge lets an external coordinator list sessions, start worktree/tmux-oriented sessions, queue bounded follow-up prompts, read status/tail/artifacts, handle structured questions, and write coordination reports without scraping terminal scrollback.

Standard setup

Use gjc setup hermes to render or install a portable MCP setup package for any controller that accepts Hermes-compatible MCP config. The default mode is render-only and writes no files.

gjc setup hermes --root /path/to/repo --profile my-bot --repo gajae-code

To install into a Hermes profile, opt in to mutation classes and pass an install target:

gjc setup hermes \ --root /path/to/repo \ --profile my-bot \ --repo gajae-code \ --mutation sessions,questions,reports \ --profile-dir /path/to/hermes/profile \ --install

The generated setup is model-agnostic and worktree-isolated. By default it renders GJC_COORDINATOR_MCP_SESSION_COMMAND as gjc --worktree, so spawned sessions launch inside a GJC-managed sibling worktree while GJC still records the original repo as the project identity for tmux/session resume. Provider/model examples are examples only; GJC does not hard-code any provider as the Hermes default. Run a non-mutating smoke check with gjc setup hermes --root /path/to/repo --smoke.

Safety model

The bridge is read-only and fail-closed by default.

export GJC_COORDINATOR_MCP_WORKDIR_ROOTS="/path/to/repo:/path/to/worktrees"

Mutating tools require both startup opt-in and per-call consent:

export GJC_COORDINATOR_MCP_MUTATIONS="sessions,questions,reports"
  • Every mutating MCP call must also include allow_mutation: true. Missing startup opt-in or per-call consent returns an error instead of falling back to shell or terminal relay.
  • Workdir roots, repo identity, safe session tokens, and tmux target liveness are validated before state writes.
  • Artifact reads are canonicalized, symlink escapes are rejected, and returned content is byte-capped by GJC_COORDINATOR_MCP_ARTIFACT_BYTE_CAP.
  • Optional namespace variables GJC_COORDINATOR_MCP_PROFILE and GJC_COORDINATOR_MCP_REPO prevent cross-profile/cross-repo enumeration; a missing namespace never widens into global session enumeration.

gjc setup hermes renders GJC_COORDINATOR_MCP_WORKDIR_ROOTS with the host platform path delimiter (: on POSIX, ; on Windows). Prefer the GJC-native worktree command over creating an unmanaged Hermes-side worktree, so session listing/resume can still group the session under the source project.

Tool surface

Read tools: gjc_coordinator_list_sessions, gjc_coordinator_read_status, gjc_coordinator_read_tail, gjc_coordinator_list_questions, gjc_coordinator_list_artifacts, gjc_coordinator_read_artifact, gjc_coordinator_read_coordination_status, gjc_coordinator_read_turn, gjc_coordinator_await_turn, gjc_coordinator_watch_events.

Mutating tools: gjc_coordinator_start_session, gjc_coordinator_register_session, gjc_coordinator_send_prompt, gjc_coordinator_submit_question_answer, gjc_coordinator_report_status, gjc_delegate_plan, gjc_delegate_execute, gjc_delegate_team.

Use gjc_coordinator_register_session to attach an already-visible tmux-backed GJC pane as the coordinator-authoritative session instead of creating a hidden gjc-coordinator-* session. The gjc_delegate_* tools are covered in Path 3.

Coordinator turn model

Treat turns, not terminal scrollback, as the unit of work. The bridge stores durable turn/question/report state and exposes bounded polling snapshots.

  1. Call gjc_coordinator_start_session with allow_mutation: true.
  2. Call gjc_coordinator_send_prompt with allow_mutation: true.
  3. Store the returned turn_id.
  4. Poll gjc_coordinator_read_turn, or call bounded gjc_coordinator_await_turn, until the turn is terminal.
  5. If gjc_coordinator_list_questions shows a question for that turn, answer with gjc_coordinator_submit_question_answer.
  6. Use gjc_coordinator_report_status with session_id and turn_id to write explicit completion/failure evidence (status: "cancelled" for policy cancellation; status: "failed" plus blocker for provider/tool/task failures).

A session may have only one active turn by default; a second prompt is rejected with active_turn_exists unless the caller passes queue: true or force: true. Queued turns are durable and promoted when the active turn reaches a terminal report. force supersedes and audits the previous active turn.

Event watch surface

gjc_coordinator_watch_events is the long-poll event journal surface — a bounded read tool, not an unbounded stream. Inputs are after_seq (default 0), optional session_id/event_types, timeout_ms capped at 30000, and limit capped at 100. The response includes events, latest_seq, timed_out, and explicit transport metadata transport: { "mcp": "long_poll", "push_subscriptions": false }. Persist latest_seq and resume safely after restart. A timeout with no events is not failure. The coordinator MCP bridge is a durable polling/await surface; it does not expose push subscriptions, so poll gjc_coordinator_read_coordination_status, gjc_coordinator_read_turn, or bounded gjc_coordinator_await_turn.

Controller config snippet

{ "mcp_servers": { "gjc_coordinator": { "command": "gjc", "args": ["mcp-serve", "coordinator"], "env": { "GJC_COORDINATOR_MCP_WORKDIR_ROOTS": "/path/to/repo", "GJC_COORDINATOR_MCP_PROFILE": "team-a", "GJC_COORDINATOR_MCP_REPO": "project", "GJC_COORDINATOR_MCP_SESSION_COMMAND": "gjc --worktree" }, "enabled": true } } }

Smoke-check the contract with gjc mcp-serve coordinator --check --json. Expect ok: true, server name gjc-coordinator-mcp, and the GJC-named tool list.

Path 3 · The gjc-delegation skill

The generated gajae-code plugin exposes the coordinator MCP server plus a host-facing skill so a host agent can delegate whole workflows to GJC and receive durable turn status plus artifacts. Each delegate tool starts (or reuses) a session and sends one workflow-tagged turn that runs a GJC skill to completion.

# Claude Code /plugin marketplace add ./plugins # Codex codex plugin marketplace add ./plugins

Then install the gajae-code plugin. The plugin files are generated by bun run generate-plugins from the coordinator contract; do not hand-edit them.

ToolGJC skillPurpose
gjc_delegate_plan/skill:ralplanDelegate consensus planning to a pending-approval plan.
gjc_delegate_execute/skill:ultragoalDelegate execution to completion with verification.
gjc_delegate_team/skill:teamDelegate parallel team execution with internal tmux workers.

Pass cwd and task; set allow_mutation: true only when the sessions mutation class is enabled and the user has approved changes. Each delegate returns a durable turn_id and emits a delegation.started event. Poll the turn_id with gjc_coordinator_await_turn or watch gjc_coordinator_watch_events — turn state is the source of truth, not terminal scrollback. Prefer these over manual start_session + send_prompt when delegating a whole workflow; drop to the manual tools only for fine-grained control they do not cover.

Fail-closed by default: the bundled MCP config sets GJC_COORDINATOR_MCP_WORKDIR_ROOTS to the host project directory and does not set GJC_COORDINATOR_MCP_MUTATIONS. Delegation is read-only until the user enables a mutation class and passes allow_mutation: true per call. GJC_COORDINATOR_MCP_REPO is a namespace label only, never a filesystem path.

Operator core loop

For a Hermes-style coordinator operating the bridge directly:

  1. Find an existing session with list_sessions, or start_session when a new session is required and mutation is enabled.
  2. Send exactly one bounded task prompt with send_prompt and store the returned turn_id.
  3. Prefer watch_events with the stored latest_seq for event-driven progress; fall back to read_turn/await_turn until terminal. If a second task is needed while one turn is active, pass queue: true.
  4. If GJC asks a structured question, list with list_questions and answer with submit_question_answer.
  5. Use report_status for coordinator-visible status and final reports; use read_tail only as advisory debug output.

Do not report completion to the user until the GJC turn is terminal, and do not infer completion from terminal scrollback alone.

Path 4 · RPC host-tool boundary

When Hermes/OpenClaw needs GJC to call the host's own tools, MCP servers, and skills, the supported boundary is RPC mode — not direct imports from GJC's runtime MCP implementation. Use @gajae-code/coding-agent/modes:

  • RpcClient to spawn and drive gjc --mode rpc.
  • defineRpcClientTool() and RpcClientOptions.customTools to expose host-owned tools.
  • RpcClient#setCustomTools() to refresh the host tool list after the host reloads capabilities.
import { RpcClient, defineRpcClientTool } from "@gajae-code/coding-agent/modes"; const client = new RpcClient({ cwd: repoPath, customTools: [ defineRpcClientTool({ name: "openclaw_skill_search", description: "Search OpenClaw skills visible to this session", parameters: { type: "object", properties: { query: { type: "string" } }, required: ["query"], additionalProperties: false, }, async execute(args, context) { context.sendUpdate("Searching OpenClaw skill registry…"); return await searchOpenClawSkills(String(args.query)); }, }), ], }); await client.start(); await client.promptAndWait("Use the host skill search when it helps.");

Treat MCP as a host implementation detail: the host discovers its MCP servers and skills, converts selected capabilities into RPC customTools, GJC calls them through host_tool_call frames, and the host returns host_tool_result. This keeps host credentials, approvals, and policy out of GJC. Do not import quarantined paths such as @gajae-code/coding-agent/runtime-mcp, /mcp, /capability/mcp, /config/mcp-schema, or /discovery/mcp-json. Good first host tools are small and policy-preserving: openclaw_skill_search, openclaw_skill_read, openclaw_mcp_call, hermes_route_message.

Verification checklist

  • The recommended routed session reaches TUI readiness and a skill prompt produces a real work signal (tool call, diff, test, or PR URL) — not just visible scrollback.
  • gjc mcp-serve coordinator --check --json returns ok: true with the GJC-named tool list.
  • A read-only call (e.g. gjc_coordinator_list_sessions) succeeds before any mutation class is enabled.
  • A mutating call without allow_mutation: true (or without startup opt-in) fails closed instead of relaying to shell.
  • A delegate tool returns a durable turn_id and emits delegation.started; the terminal turn state — not scrollback — confirms completion.
  • For RPC: GJC emits host_tool_call, the host returns host_tool_result, and direct imports from quarantined MCP paths still fail. See packages/coding-agent/test/rpc-host-tools.test.ts.
